# IBM TM1 规则语言

> **极度小众警告**：这是一门全球可能只有几千人熟练掌握的技术

IBM TM1（现归属于 IBM Planning Analytics）是一款面向 **多维建模、财务计划、预算预测、业务分析** 的内存 OLAP 工具，而 **TM1 Rule（规则）** 是其核心计算引擎，用于在多维立方体中定义 **动态计算、数据联动、权限控制、逻辑映射** ，属于偏企业级 BI/绩效系统的小众专业技术。

---

## 快速了解：30秒抓住 TM1 的感觉

### 一句话定义
TM1 是财务人员的 Excel steroids — — 把 Excel 的单元格计算扩展到企业级的多维数据立方体。

### 使用场景想象
想象你在管理一家跨国公司的预算：
- 100 个子公司 × 50 个部门 × 12 个月 × 100 个科目 = **600 万个单元格**
- 汇率变动、部门间分摊、合并抵消... 规则复杂到 Excel 直接崩溃
- TM1 把这些装进**内存里的多维立方体**，规则一改，全公司数据秒级更新

### 核心感觉
```
Excel 思维：A1 = B1 * C1
TM1 思维：['销售额', '中国', '2026', 'Q1'] = ['销量'] * ['单价']
```
维度切片替代了单元格引用，这就是 TM1 的精髓。

### 有多小众？
| 对比项 | Python | TM1 |
|--------|--------|-----|
| 全球开发者 | 1500万+ | ~1万 |
| Stack Overflow 问题 | 200万+ | <500 |
| 中文教程 | 数十万 | 几乎为零 |
| 企业用户 | 无数 | ~5000家 |

**结论**：这是一门只在世界500强财务部门存在的"幽灵技术"。

## 一、TM1 Rule 是什么？

- **作用**：在 TM1 Cube（多维立方体）中，**不存储计算结果，而是实时动态计算**

- **特点**：
  - 基于维度交集定义计算逻辑
  - 实时计算，修改规则立即生效
  - 支持条件、循环、引用、函数、维度过滤
  - 常用于：比率计算、分摊、汇率换算、合并报表、校验逻辑

- **与 TI（TurboIntegrator）区别**：
  - **Rule**：动态查询时计算，不写库
  - **TI**：批量 ETL，写入数据到 Cube 单元格

## 二、基本语法结构

TM1 Rule 语法非常精简，典型结构如下：

```plain
[维度筛选条件] = N: 计算公式 ;
[维度筛选条件] = C: 汇总计算公式 ;
```

### 关键字说明

| 关键字 | 说明 |
|--------|------|
| `[]` | 定位单元格（切片 / 交集） |
| `N:` | 针对**基础叶节点（N 级）** 计算 |
| `C:` | 针对**汇总节点（C 级）** 计算 |
| `S:` | 字符串类型计算 |
| `DB()` | 跨 Cube 取数核心函数 |
| `[] = []` | 单元格之间直接映射 |
| `STET` | 停止规则，继承原始值 |
| `CONTINUE` | 继续执行下一条规则 |

---

## 三、最常用核心语法示例

### 1. 简单计算

```plain
['销售额'] = ['销量'] * ['单价'];
```

### 2. 区分 N 级 / C 级

```plain
['毛利率'] = N: (['销售额'] - ['成本']) / ['销售额'];
['毛利率'] = C: ['销售额'] -> ['毛利率'];
```

### 3. 条件判断（IF）

```plain
['折扣'] = IF(['销量'] > 1000, 0.1, 0.05);
```

### 4. 跨 Cube 取数（DB 函数）

```plain
['汇率'] = DB('汇率Cube', !年, !币种, '即期汇率');
```

> `!维度名` 表示**当前维度当前元素**，是 Rule 最常用的引用方式。

### 5. 维度过滤

```plain
['部门':'销售部', '费用'] = ['总费用'] * 0.6;
```

### 6. 跳过规则（STET）

```plain
['特殊产品'] = STET;
```

表示该单元格不使用规则，保留输入值。

### 7. 字符串规则（S:）

```plain
['备注'] = S: '自动计算';
```

## 四、常用内置函数（高频）

| 函数 | 功能 |
|------|------|
| `DB()` | 跨 Cube 取值 |
| `IF()` | 条件判断 |
| `ABS()` / `ROUND()` | 数值处理 |
| `SUM()` / `AVG()` | 聚合 |
| `ATTRS()` / `ATTRN()` | 读取维度属性（文本/数字） |
| `ELLEV()` | 获取维度层级 |
| `DIMIX()` | 判断元素是否存在 |
| `EXP` / `LOG` | 财务计算函数 |

## 五、Rule 执行顺序与注意事项

1. **自上而下执行**，后面规则可覆盖前面
2. 尽量避免**循环引用**（A 依赖 B，B 依赖 A）
3. 复杂逻辑过多会**影响查询性能**
4. 汇总规则（C:）写不好容易出现**汇总错误**
5. Rule 只影响查询，不修改原始存储数据

## 六、典型业务场景用法

### 场景 1：汇率换算

```plain
['本币金额'] = ['原币金额'] * DB('汇率', !日期, !币种, '月末汇率');
```

### 场景 2：部门费用分摊

```plain
['人力成本分摊'] = ['总人力成本'] * ['分摊比例'];
```

### 场景 3：报表合并（抵消分录）

```plain
['内部交易抵消'] = -['内部收入'];
```

## 七、技术定位：OLAP 领域的特殊位置

### OLAP 三种架构对比

| 架构 | 代表产品 | 特点 | TM1 位置 |
|------|----------|------|----------|
| **MOLAP** | TM1, SSAS | 多维立方体，内存计算 | **先驱者** |
| **ROLAP** | SAP BW, Oracle BI | 关系型数据库扩展 | 不同赛道 |
| **HOLAP** | 混合方案 | 部分内存，部分磁盘 | TM1 后期支持 |

### TM1 的技术独特性

1. **内存计算先驱**
   - 1984年就实现了全内存计算
   - 比 SAP HANA 早 25 年
   - 比主流内存数据库概念早 20 年

2. **实时规则引擎**
   - 规则修改立即生效，无需重新计算
   - 查询时动态计算，不占用存储空间
   - 支持复杂的多维交叉计算

3. **Excel 原生集成**
   - 财务人员的"第二大脑"
   - 可以直接在 Excel 中操作多维数据
   - 降低了财务人员的接受门槛

## 八、适用范围：谁在用 TM1？

### 典型用户画像

| 维度 | 描述 |
|------|------|
| **企业规模** | 年收入 10 亿美元以上的大型企业 |
| **行业分布** | 金融、制造、零售、能源、医药 |
| **部门** | 财务部、FP&A（财务规划与分析）、预算部 |
| **地区** | 北美、欧洲为主，中国较少 |

### 典型应用场景

1. **全面预算管理（EPM）**
   - 年度预算编制
   - 滚动预测
   - 多版本情景分析

2. **财务报表合并**
   - 跨国企业多币种合并
   - 内部交易抵消
   - 会计准则转换（GAAP/IFRS）

3. **销售与运营计划（S&OP）**
   - 需求预测
   - 供应链优化
   - 产能规划

4. **盈利能力分析**
   - 产品/客户/渠道盈利分析
   - 成本分摊
   - 定价策略支持

### 中国市场的特殊性

- **使用企业少**：主要集中在跨国企业在华分支
- **本土替代品**：用友、金蝶、元年等本土 EPM 厂商
- **实施成本高**：需要 IBM 认证顾问，人力成本极高
- **人才稀缺**：全国 TM1 顾问估计不超过 **200 人**

## 十、为什么 TM1 如此小众？

### 1. 市场定位狭窄

```
全球企业软件市场
├── ERP (SAP, Oracle) ──────────── 市场巨大
├── CRM (Salesforce) ───────────── 市场巨大
├── BI 分析 (Tableau, Power BI) ── 市场大
├── 数据库 (Oracle, SQL Server) ── 市场巨大
└── EPM/预算 (TM1, Anaplan) ────── 市场极小 ⬅ TM1 在这里
```

- EPM 市场本身只占企业软件的 **2-3%**
- TM1 在 EPM 市场占比约 **15-20%**
- 全球活跃企业用户估计 **3000-5000 家**

### 2. 生态封闭昂贵

| 成本项 | 说明 |
|--------|------|
| **License** | 按 CPU 核心或用户数收费，动辄百万级 |
| **实施费用** | IBM 认证顾问 3000-5000 元/人天 |
| **维护费用** | 年维护费为 License 的 18-22% |
| **硬件要求** | 需要大内存服务器（TB 级） |

- 中小企业完全无法承受
- 只有大型跨国企业才用得起

### 3. 学习资源匮乏

- **官方文档**：IBM Knowledge Center，晦涩难懂
- **中文资料**：几乎为零
- **社区论坛**：IBM Community，活跃度低
- **Stack Overflow**：相关问题 < 500 个
- **GitHub**：开源项目几乎为零

对比 Python：
- Python 相关问题：200万+
- Python 开源项目：数百万
- Python 中文教程：数十万篇

### 4. 技术栈封闭

- 无法本地免费使用（无社区版）
- 无法开源贡献
- 无法与其他技术栈深度集成
- 学习后技能难以迁移

### 5. 业务门槛高

- 需要懂**财务知识**（预算、合并报表、会计准则）
- 需要懂**业务场景**（S&OP、盈利能力分析）
- 纯技术人员难以独立完成项目

## 十一、AI 时代下的价值评估

### TM1 会被 AI 取代吗？

**短期（3-5年）：不会**

- 大型企业的财务系统迁移成本极高
- AI 目前擅长的是"分析"，不是"规则定义"
- TM1 的核心是业务规则建模，不是数据处理

**长期（10年+）：可能边缘化**

- AI 可能实现自然语言到规则的自动转换
- 云原生 EPM 工具（如 Anaplan）更具扩展性
- 但遗留系统维护需求会持续存在

### 工业工程架构师应该学吗？

####  适合学习的情况

1. **你在大型制造企业做财务数字化**
   - 企业已有 TM1 系统
   - 需要维护或优化现有模型

2. **你专注 EPM/预算领域**
   - 想成为财务数字化专家
   - 愿意深耕细分领域

3. **你有财务背景 + 技术兴趣**
   - 财务 BP 转型
   - 会计 + 编程复合背景

####  不适合学习的情况

1. **你追求技术开放生态**
   - TM1 技能难以迁移
   - 会限制你的技术视野

2. **你在中小企业工作**
   - 根本用不起 TM1
   - 学了也没地方用

3. **你想快速切换技术栈**
   - TM1 学习曲线陡峭
   - 投入产出比低

### 综合评价

| 维度 | 评分 | 说明 |
|------|------|------|
| **市场需求** | 2 | 需求稳定但范围极窄 |
| **薪资水平** | 4 | 稀缺性带来溢价 |
| **技术前景** | 2 | 云化和 AI 可能边缘化 |
| **学习难度** | 3 | 语法简单，业务复杂 |
| **生态开放** | 1 | 完全封闭 |
| **职业迁移** | 1 | 技能难以迁移 |

**结论**：TM1 是一门**高价值但高风险**的技术。如果你已经在相关领域，值得深入；如果想转型进入，需谨慎评估。
